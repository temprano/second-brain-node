OpenClaw · Webhook Integration

Two-way
OpenClaw ↔ Pipeline

Wire OpenClaw so it triggers your pipeline on `research:` / `learning:` messages, and the pipeline notifies you back when complete.

The two-way flow
You (WhatsApp/Telegram)
→
OpenClaw agent
→
POST /run
→
Your VPS webhook server
→
Pipeline runs
Pipeline completes
→
POST /hooks/agent
→
OpenClaw agent
→
You get a message back
💡Direction 1: you trigger the pipeline by messaging OpenClaw. Direction 2: the pipeline posts back to OpenClaw when done so you get a completion message — no polling, no checking job status manually.
PART A Configure OpenClaw to trigger the pipeline
STEP 01Enable webhooks in OpenClaw config.yaml

This enables OpenClaw to both receive callbacks from your pipeline and send requests out. Edit ~/.openclaw/config.yaml (or wherever your config lives):

# ~/.openclaw/config.yaml — add this hooks section hooks: enabled: true # Token for inbound hooks — pipeline uses this to POST back to you # Must match OPENCLAW_HOOKS_TOKEN in your VPS .env token: "${OPENCLAW_HOOKS_TOKEN}" # URL path — default is /hooks path: "/hooks" # Keep pipeline notifications in their own session thread defaultSessionKey: "hook:second-brain" # Only allow the main agent to be triggered allowedAgentIds: ["main"] # Keep session key overrides off allowRequestSessionKey: false
⚠ After editing config.yaml, restart OpenClaw: pm2 restart openclaw (or however you run it)
STEP 02Add the Second Brain skill to SOUL.md

Open your SOUL.md (usually ~/.openclaw/SOUL.md) and paste in the snippet from OPENCLAW_SOUL_SNIPPET.md. This tells the agent which message patterns to intercept and what to do with them.

Key part of the snippet — the agent looks for these prefixes and fires the webhook immediately:

# Excerpt from SOUL.md snippet Trigger when the user message starts with any of: research: — find best options, compare, factual research learning: — deep understanding and skill building study: — structured facts and definitions review: — book/product/tool summaries news: — latest developments on a topic When triggered, POST to your pipeline immediately: POST {{SECOND_BRAIN_WEBHOOK_URL}}/run X-API-Key: {{SECOND_BRAIN_SECRET}} { "command": "<full user message>" }

Replace SECOND_BRAIN_WEBHOOK_URL with your VPS address (e.g. http://your-vps-ip:3579) and SECOND_BRAIN_SECRET with your WEBHOOK_SECRET from .env.

💡The soul snippet file is included in your project as OPENCLAW_SOUL_SNIPPET.md. Open it, copy the contents, paste into your SOUL.md.
PART B Configure the VPS pipeline to notify OpenClaw back
STEP 03Add OpenClaw connection vars to your VPS .env

These tell the pipeline where to POST the completion notification when a job finishes.

# Add to your VPS .env # Where OpenClaw is running # Same VPS as pipeline: http://localhost:18789 # OpenClaw on Windows PC: http://<tailscale-ip>:18789 OPENCLAW_HOST=http://localhost:18789 # Must match hooks.token in OpenClaw config.yaml # Generate: node -e "console.log(require('crypto').randomBytes(32).toString('hex'))" OPENCLAW_HOOKS_TOKEN=your-openclaw-hooks-token-here # Which agent to notify (default: main) OPENCLAW_AGENT_ID=main # Set false to disable completion notifications OPENCLAW_NOTIFY=true
⚠ Both OPENCLAW_HOOKS_TOKEN values must match exactly — the one in OpenClaw's config.yaml and the one in your VPS .env. This is the shared secret that authenticates the pipeline's callback.
STEP 04Restart the pipeline server after .env changes
# On your VPS: pm2 restart second-brain pm2 logs second-brain --lines 10 # confirm it started cleanly
PART C Test the full two-way flow

Send one of these to OpenClaw on WhatsApp/Telegram/Discord. The agent should detect the prefix and fire the pipeline immediately.

STEP 06Test outbound: manually trigger a callback

Test the pipeline-to-OpenClaw notification directly from your VPS terminal to confirm the token and host are correct:

# On your VPS — test the outbound notification curl -X POST http://localhost:18789/hooks/agent \ -H 'Authorization: Bearer YOUR_OPENCLAW_HOOKS_TOKEN' \ -H 'Content-Type: application/json' \ -d '{"message":"✓ Pipeline test — webhook connection working!","agentId":"main","wakeMode":"now","deliver":true}' # You should receive a message on WhatsApp/Telegram within seconds # If you get 401: token mismatch — check both sides match # If connection refused: OpenClaw not running, or wrong port
ALSO Keep your existing message triggers as a fallback
NOTEMessage triggers and webhooks work side by side

You don't have to choose. Message triggers (what you're already using) let the agent handle the command itself inline. Webhooks offload heavy async work to your VPS pipeline so the agent doesn't block waiting for results.

The difference in practice:

# Message trigger approach (existing) # Agent processes inline — simpler but slower, uses agent context window You: "research quantum computing" Bot: [thinks for 30s] Here's what I found... [pastes everything into chat] # Webhook approach (new) # Agent fires VPS pipeline — async, fast ack, full pipeline runs in background You: "research: quantum computing" Bot: [immediate] Queued! ~45 seconds... [45s later] Done! Note in Obsidian, 8 cards in Anki.
💡The colon suffix (research: vs research) is a clean way to route: no colon = agent handles it inline, colon = fire the full pipeline. You can adjust this trigger pattern in SOUL.md however makes most sense to you.
